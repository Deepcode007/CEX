CEX -- Centralized Exchange: Complete Codebase Analysis

1. PROJECT OVERVIEW

CEX is a centralized cryptocurrency/asset exchange built as a microservice architecture. It simulates a real exchange with an order book matching engine, wallet management, user authentication, and asynchronous database persistence. The project is a monorepo (single git repo) containing four independent services/packages, each with its own package.json, Prisma schema, and .env.

Redis Communication Pattern (Request/Response):
1. Backend generates a correlationId (UUID) and creates a Promise stored in pendingResponses Map
2. Backend lPushes JSON to queue1_requests containing {id, responseQueue: "be1", type, order}
3. Backend listener brPops on "be1" queue for responses
4. When a response arrives with matching id, the stored Promise is resolved
5. If no response arrives within ENGINE_TIMEOUT_MS, the Promise is rejected

- curr_Users: Map<string, Orders[]>   -- Tracks orders per user
- BALANCES: Map<string, UserBalances> -- User balances cached in memory

OrderBook Class:
- bids: BTree sorted descending, highest price first
- asks: BTree sorted ascending, lowest price first
- orderMap: Map<string, Orders> for O(1) order lookup
- addOrder(), processOrder(), executeTrade(), cancelOrder(), getDepth()
- getBestBid() / getBestAsk() for top-of-book access
Matching Engine Flow:
New Order
  |
  v
Lock Balance                 Deduct available balance and move funds to locked
  |
  v
processOrder()
  |-- No match  --> Add to book as a resting order
  |
  +-- Match found
       |
       v
executeTrade()               FIFO within the matched price level
       |
       |-- Update maker balance
       |-- Update taker balance
       |-- Stream four PROCESS_ORDER events to DB_DUMP_STREAM
       |   |-- Maker base asset transaction
       |   |-- Maker quote asset transaction
       |   |-- Taker base asset transaction
       |   +-- Taker quote asset transaction
       +-- Return fill details

Cancellation:
- Remove order from the BTree price level
- Unlock frozen balance
- Stream CANCEL_ORDER to UpdateWorker

Price-Time Priority:
- Price: best price matched first
- Time: FIFO within the same price level

Supported Order Types:
- limit  -- specify price + quantity
- market -- quantity only, matched at best available price

4.3. UpdateWorker/ -- Database Persistence Worker

Purpose: Asynchronous database writer. Consumes events from the DB_DUMP_STREAM Redis stream and persists order/trade/balance changes to PostgreSQL. This decouples the hot path from slow database writes.

Language/Framework: TypeScript, Bun runtime (no HTTP framework -- pure Redis stream consumer)

Entry Point: /Users/deep/Desktop/Projects/CEX/UpdateWorker/src/server.ts

Stream Consumption Pattern:
- Uses xRead with BLOCK: 0 on stream DB_DUMP_STREAM
- Reads up to 50 messages per batch
- Tracks lastId to resume from the last processed message
- Starts from "$" so only new messages are consumed after connection time

Message Types Handled:
CREATE_ORDER       prisma.order.create()
PROCESS_ORDER      Transaction: create transaction + update wallet + create fill + update order status
DEPOSIT/WITHDRAW   Transaction: update wallet balance + create transaction record
CANCEL_ORDER       Update order status to "cancelled" + refund wallet balance

7. INTER-SERVICE COMMUNICATION

Channel 1: Backend <> OrderBook (Redis List Queues -- Request/Response)
Backend   -> OrderBook    LPush / brPop  queue1_requests     JSON {id, responseQueue, type, order}
OrderBook -> Backend      rPush / brPop   be1                 JSON {id, success, data/error}

This is a synchronous request-response over async queues. The backend generates a correlation UUID,
pushes to the request queue, and blocks on a Promise that resolves when the named response queue
receives a message with the matching ID. Timeout is 30 seconds.

Supported command types: create_order, get_depth, get_user_balance, get_order, cancel_order, deposit, withdraw

Channel 2: OrderBook -> UpdateWorker (Redis Stream -- Fire-and-Forget)
OrderBook pushes events as trades execute. UpdateWorker consumes them asynchronously in batches of up to 50.
Message types: CREATE_ORDER, PROCESS_ORDER, DEPOSIT, WITHDRAW, CANCEL_ORDER.

Channel 3: All Services -> PostgreSQL (Direct via Prisma)
- Backend: Reads users, stocks, orders, fills, wallets directly from DB
- OrderBook: Reads wallet balances on cache miss
- UpdateWorker: Writes all persistent state

9. CONFIGURATION AND DEPLOYMENT

Running Locally:

Each service is started independently:

cd backend && bun run src/server.ts       # HTTP API on port 3000
cd OrderBook && bun run src/server.ts     # Matching engine (no HTTP port)
cd UpdateWorker && bun run src/server.ts  # DB writer (no HTTP port)

Infrastructure Requirements:

- PostgreSQL on localhost:5432 with database CEX
- Redis on localhost:6378 (non-standard port; used by all services)
- Stocks (SOL, ETH, USD) must be pre-seeded in the Stock table

No Docker/K8s:

There are no Docker, docker-compose, Kubernetes, or CI/CD configuration files in the project. Deployment is manual/local only.

Horizontal Scaling Note:

The backend_Id env variable (be1) is the response queue name, which means multiple backend instances could theoretically run with different IDs (e.g., be1, be2), each listening on its own response queue. However, the OrderBook engine is a single process with in-memory state, so it cannot be horizontally scaled without architectural changes.

10. KEY ARCHITECTURAL OBSERVATIONS

1. CQRS-lite Pattern: The OrderBook engine is the single source of truth for real-time state
   (balances, order books). The database is updated asynchronously by the UpdateWorker and is used
   primarily for historical queries and cold-start data loading.

2. In-Memory Engine: The entire order book and user balances live in memory within the OrderBook process.
   This gives extremely fast matching but means all state is lost on restart. There is no visible
   snapshot/recovery mechanism.

3. Event Sourcing (partial): Trade events are streamed via Redis Streams (DB_DUMP_STREAM), which provides
   ordered persistent event delivery to the UpdateWorker. However, there is no replay mechanism to
   rebuild engine state from events.

4. Shared Schema, Separate Codegen: All three services have identical schema.prisma files but generate
   their own Prisma client into local prisma/generated/ directories. Only the backend/ has migration files.

5. Cross-Service Import (code smell): The OrderBook class imports across service boundaries to share env
   config and generated Prisma enums. These imports create tight coupling between packages.

6. Balance Types: Balances use Int, suggesting the system works with whole units (cents, satoshis, or
   similar atomic units) to avoid floating-point precision issues.
